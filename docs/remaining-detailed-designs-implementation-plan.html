Curriculum-to-Learning Golden Path

This exclusion is the minimal end-to-end chain from a valid curriculum package to an assigned learner opening content and retaining progress. These 11 designs and their 27 L2 requirements remain owned by the active golden-path implementation. Other waves may consume their public contracts, but shall not duplicate or replace their implementations.

Integration rule: when the golden path merges, record its API, event, persistence, authorization, and UI-route seams as the baseline. Rebase remaining work onto that baseline and resolve deviations through an ADR instead of parallel implementations.

Execution model

How to use this plan

  • Implement one linked feature design as one reviewable vertical slice whenever practical.
  • Start a slice only when its “Starts when” condition and unresolved decisions are cleared.
  • Put every owned L2 identifier in tests, telemetry, and the pull-request description.
  • Keep frontend, API, application, domain, persistence, security, and operations changes together.
  • Use feature flags for incomplete journeys; never expose a partially authorized route.
  • Complete each wave’s exit gate before downstream waves depend on its contracts.

Every slice

Definition of done

  • All L2 acceptance scenarios owned by the feature are automated at the lowest reliable layer and mapped by identifier.
  • Tenant, role, scope, lifecycle-state, replay, concurrency, and failure-path tests cover the feature’s trust boundaries.
  • The vertical slice includes required Angular UI, API, application/domain behavior, persistence, migrations, and provider adapters.
  • Structured telemetry, audit evidence, client-safe errors, redaction, and correlation are implemented without protected payloads.
  • Keyboard, screen-reader semantics, 200% zoom, narrow viewport, reduced motion, capability fallback, and performance budgets are checked where applicable.
  • API/event/schema compatibility is versioned; migrations and rollback or forward-fix steps are documented and exercised.
  • Operational ownership, alert/runbook changes, feature-flag behavior, and deployment evidence accompany the pull request.
  • The linked detailed design and affected requirement documents are updated if implementation decisions change their contract.
