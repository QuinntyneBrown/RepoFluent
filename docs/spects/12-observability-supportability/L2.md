## Detailed requirements

### L2-OBS-01 — Telemetry event envelope

**Traces to:** L1-OBS-01, L1-OBS-02

Structured events shall use a versioned envelope containing event name/version, timestamp, environment, service/component, outcome, safe correlation/trace/span identifiers, tenant pseudonymous or approved identifier, actor type where safe, operation/entity type, duration/error category, and schema version. Payload size and cardinality shall be bounded.

**Acceptance criteria**

- **GIVE** one successful and one failed operation from each required domain
- **WHEN** telemetry contract tests inspect emitted events
- **THEN** required envelope fields and outcome-specific safe fields validate, identifiers correlate, and prohibited/unbounded fields are absent

### L2-OBS-02 — Domain telemetry coverage

**Traces to:** L1-OBS-01

Instrumentation shall cover authentication start/result, authorization decision, package receipt/scan/validation/import, renderer success/failure/fallback, progress receive/persist, attempt start/save/submit/grade, lifecycle review/publish/retire, export, and retention workflow. Each domain shall define success, expected rejection, dependency failure, and internal failure categories.

**Acceptance criteria**

- **GIVE** the domain conformance scenario set
- **WHEN** each scenario is executed in an instrumented environment
- **THEN** expected telemetry appears once with correct category/result and can be joined to its request/job/audit evidence

### L2-OBS-03 — End-to-end correlation

**Traces to:** L1-OBS-02

An inbound request shall accept or create a safe correlation/trace identifier and propagate it through synchronous calls, queued work, domain events, audit references, logs, and safe client error response. Untrusted incoming identifiers shall be validated/bounded and shall not replace security ownership context.

**Acceptance criteria**

- **GIVE** an upload triggers scan, validation, draft import, audit, and status update across processes
- **WHEN** support queries using the client-visible correlation identifier
- **THEN** every stage can be traced in order without using package contents or ambiguous text search

### L2-OBS-04 — Import and lifecycle monitoring

**Traces to:** L1-OBS-03

Dashboards/alerts shall track package intake volume, rejection by category, validation failure/warning rate, processing latency, queue age, job success/retry/failure, preview/publication failures, and stale jobs, segmented safely by version/environment and tenant only where authorized.

**Acceptance criteria**

- **GIVE** a processing job exceeds the configured stale threshold and import failure rate exceeds its alert condition
- **WHEN** monitoring evaluates the signals
- **THEN** an actionable alert identifies environment/component/category, impact, correlation examples, owner, and runbook without customer payload

### L2-OBS-05 — Learner rendering and performance monitoring

**Traces to:** L1-OBS-03, L1-OBS-08

Real-user and synthetic monitoring shall measure learner-shell usability, interaction latency, renderer errors, fallback/GPU capability, long tasks, and critical route availability under the defined browser/profile. Metrics shall support p75 budget evaluation and release/version comparison.

**Acceptance criteria**

- **GIVE** production traffic with one renderer regression and one browser using GPU fallback
- **WHEN** performance/error dashboards aggregate the release
- **THEN** the regression is attributable to route/release/component, p75 budgets are calculable, and fallback is measured without treating it as failure when core behavior succeeds

### L2-OBS-06 — Assessment reliability monitoring

**Traces to:** L1-OBS-03, L1-OBS-06

Monitoring shall track attempt start/save/submit/grade success, latency, retries, idempotency replays, conflicts, timeouts, feedback-release failures, and stuck grading work. Alerts shall prioritize risk of learner response/evidence loss or duplicate grading.

**Acceptance criteria**

- **GIVE** assessment submission persistence begins failing above the defined threshold
- **WHEN** monitoring detects the failure
- **THEN** on-call receives an actionable alert before silent evidence loss, impacted correlations are queryable, and protected responses are not included

### L2-OBS-07 — Progress-write idempotency

**Traces to:** L1-OBS-06

Progress commands shall carry a client event/idempotency identifier and version context, persist atomically, deduplicate replay, and resolve out-of-order updates without regressing durable completion. The service shall return a durable acknowledgement or a retry-safe failure.

**Acceptance criteria**

- **GIVE** the same progress event is delivered repeatedly and an older position event arrives after completion
- **WHEN** all writes are processed
- **THEN** one event/effect is recorded, completion does not regress, and replay/out-of-order metrics are emitted without learner content

### L2-OBS-08 — Assessment-submission idempotency

**Traces to:** L1-OBS-06

Final submission shall be atomic and keyed to attempt plus idempotency identity. Retry shall return the existing submission/result or current processing state. A failure between persistence and response shall not create a second result or consume another attempt.

**Acceptance criteria**

- **GIVE** submission commits but the client response is lost
- **WHEN** the client retries with the same key
- **THEN** the existing immutable submission is returned/continued, grading occurs once, and reliability telemetry classifies the replay

### L2-OBS-09 — Safe logging policy

**Traces to:** L1-OBS-04

Logging libraries/configuration shall use allow-listed structured fields, centralized redaction, bounded untrusted values, protected log access, retention, and automated secret/payload conformance tests. Source, answers, responses/free text, tokens, assertions, and credentials shall be prohibited by default.

**Acceptance criteria**

- **GIVE** seeded protected values traverse success and failure paths
- **WHEN** logs/traces/metrics are captured and scanned
- **THEN** no protected value appears and sufficient identifiers/category/timing remain for diagnosis

### L2-OBS-10 — Client-safe error identifiers

**Traces to:** L1-OBS-05

Unexpected or supportable failures shall return a non-secret, non-sequential support identifier or safe correlation reference with user-appropriate recovery guidance. Internal diagnostics shall map it to component/time/category/trace. Stack traces and internal topology shall not be returned to clients.

**Acceptance criteria**

- **GIVE** an internal rendering or persistence exception
- **WHEN** the operation fails
- **THEN** the user receives safe guidance and an identifier that authorized support can resolve to diagnostics, with no stack/source/secret disclosure

### L2-OBS-11 — Tenant-safe diagnostic bundle

**Traces to:** L1-OBS-09; ADM-04

Authorized administrators/support shall generate a bounded diagnostic bundle containing product/config/schema versions, safe job/error summaries, capability state, health indicators, correlations, and timestamps. Bundle access/generation/download shall be audited and filtered to tenant scope and redaction policy.

**Acceptance criteria**

- **GIVE** a tenant administrator generates diagnostics for a failed import
- **WHEN** security and support review the bundle
- **THEN** it supports stage/version/correlation diagnosis, contains no package/source/answers/secrets/other-tenant data, and generation/download are audited

### L2-OBS-12 — Service-level objectives and ownership

**Traces to:** L1-OBS-08

Before production launch, each critical journey shall have agreed indicators/objectives, measurement source, error-budget rule or equivalent, dashboard, alert threshold, owner, and escalation. At minimum this covers authentication, learner shell, progress write, assessment submit/grade, import-to-preview, and publication.

**Acceptance criteria**

- **GIVE** the production-readiness review
- **WHEN** each critical journey is assessed
- **THEN** its SLI/SLO, measurement, owner, alert/runbook, and current evidence are present and no launch-critical journey is unowned

### L2-OBS-13 — Backup policy and coverage

**Traces to:** L1-OBS-07

Backup policy shall identify in-scope stores, encryption, schedule, retention, isolation, integrity checking, RPO/RTO targets, tenant/config/content/evidence dependencies, and handling of deleted/held data. Backup success alone shall not substitute for restore verification.

**Acceptance criteria**

- **GIVE** the production data inventory and declared recovery objectives
- **WHEN** backup coverage is audited
- **THEN** every required durable store and dependency maps to an encrypted monitored backup/control or an approved rationale, with no unidentified recovery gap

### L2-OBS-14 — Restore verification

**Traces to:** L1-OBS-07

Restore exercises shall occur on a documented cadence in an isolated environment and verify integrity, tenant boundaries, identity/configuration consistency, immutable curriculum/attempt/audit links, derived-store rebuild, and achieved recovery objectives. Results and remediation shall be retained.

**Acceptance criteria**

- **GIVE** a representative backup with multiple tenants, published versions, progress, attempts, and audit evidence
- **WHEN** the scheduled restore exercise completes
- **THEN** records and relationships reconcile, tenant isolation passes, derived data is rebuilt/validated, achieved RPO/RTO are recorded, and findings have owners

### L2-OBS-15 — Incident response readiness

**Traces to:** L1-OBS-07

The incident plan shall define detection, triage, severity, roles, containment, evidence preservation, tenant/customer/legal communication, recovery, post-incident review, and escalation for security, privacy, availability, data integrity, and protected-answer incidents. Contact paths and runbooks shall be exercised.

**Acceptance criteria**

- **GIVE** a tabletop scenario involving suspected cross-tenant disclosure and assessment-answer exposure
- **WHEN** the incident exercise runs
- **THEN** responders demonstrate detection, containment, evidence/communication decisions, recovery ownership, and track all identified readiness gaps

### L2-OBS-16 — Dependency failure and graceful degradation

**Traces to:** L1-OBS-03, L1-OBS-08

Critical dependency failure modes shall define timeout, retry/backoff, circuit behavior, idempotency, user outcome, alert, and recovery. Optional GPU/provider/notification/analytics dependencies shall not prevent safe core learning where the PRD requires fallback.

**Acceptance criteria**

- **GIVE** repository provider, WebGPU, notification delivery, and analytics pipelines are unavailable while published content storage is healthy
- **WHEN** a learner resumes core curriculum
- **THEN** core learning/progress remains usable within defined degradation, optional features show safe status, and operations receive dependency-specific signals

### L2-OBS-17 — Release operational-readiness gate

**Traces to:** L1-OBS-07, L1-OBS-08

Production release approval shall require current evidence for telemetry coverage, dashboards/alerts, performance budgets, security/privacy gates, backup/restore, incident/on-call ownership, runbooks, capacity/scale profile, dependency health, and rollback/recovery plan. Exceptions shall be approved, scoped, owned, and expiring.

**Acceptance criteria**

- **GIVE** a pilot production release candidate
- **WHEN** the operational-readiness gate executes
- **THEN** all mandatory evidence is current and passing or a formally approved time-bounded exception exists, and the release decision is recorded
