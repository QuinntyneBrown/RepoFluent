## Detailed requirements

### L2-CIC-01 — Versioned contract release unit

**Traces to:** L1-CIC-01, L1-CIC-05; CUR-01

Each contract release shall be an immutable, checksummed unit containing the JSON Schema, ICD, compatibility declaration, validation-code catalog, fixtures, and release notes. Each artifact shall identify the exact contract version and shall be retrievable without depending on a particular model or agent product.

**Acceptance criteria**

- **GIVE** a published contract version selected from the supported-version catalog
- **WHEN** an author downloads its release unit
- **THEN** all required artifacts are present, their declared versions agree, checksums validate, and the artifacts can be used offline

### L2-CIC-02 — Package metadata and source snapshot

**Traces to:** L1-CIC-02; PRD 8.3

The schema shall represent package stable identifier, contract version, title, description, ownership, locale, creation metadata, and source snapshot. A source snapshot shall support repositories, repository-relative identity, branches or commits, document identifiers, and source timestamps where available.

**Acceptance criteria**

- **GIVE** a package generated from one C# repository, one Angular repository, and approved architecture documents
- **WHEN** the package validates
- **THEN** consumers can identify the package, contract, owner, locale, and reproducible source inputs without any local absolute path

### L2-CIC-03 — Architecture and learning model

**Traces to:** L1-CIC-02

The schema shall model systems, subsystems, responsibilities, boundaries, dependencies, terminology, and typed relationships, together with courses, modules, lessons, learning objectives, prerequisites, estimated duration, difficulty, audience, tags, required/optional status, ordering, and completion rules.

**Acceptance criteria**

- **GIVE** a representative order-processing curriculum with system relationships and a multi-module course
- **WHEN** the fixture validates and is traversed by identifier
- **THEN** every ordered learning element, prerequisite, objective, system mapping, audience, and completion rule resolves unambiguously

### L2-CIC-04 — Content block vocabulary

**Traces to:** L1-CIC-02, L1-CIC-07

The schema shall use a discriminated, allow-listed content-block model for structured prose, callouts, diagrams or accessible system maps, code references, code tours, examples, glossary links, and knowledge checks. Arbitrary active HTML, script, executable content, macro content, and undeclared remote resources shall be invalid.

**Acceptance criteria**

- **GIVE** a lesson containing supported prose, callout, diagram description, and code-reference blocks
- **WHEN** the lesson validates
- **THEN** each block has a known discriminator and required accessible metadata

- **GIVE** a content block containing an embedded script or undeclared remote resource
- **WHEN** the package validates
- **THEN** validation returns a blocking safety issue at the exact block path

### L2-CIC-05 — Code references and tours

**Traces to:** L1-CIC-09; CODE-01–CODE-04

Code references shall require repository-relative paths and shall support repository identifier, optional commit, symbol, line/range anchors, language, supplied excerpt, content classification, and explanatory provenance. Code tours shall contain an ordered list of resolvable references with learner guidance at each step.

**Acceptance criteria**

- **GIVE** a multi-step tour across Angular and C# files bound to a source commit
- **WHEN** the tour validates
- **THEN** every step resolves to a declared repository and reference, preserves its order, and contains enough metadata for snapshot and drift presentation

- **GIVE** a reference containing `C:\\Users\\author\\repo\\file.cs` or another absolute author-local path
- **WHEN** the package validates
- **THEN** validation rejects it with the stable forbidden-path issue code and exact JSON path

### L2-CIC-06 — Assessment representation

**Traces to:** L1-CIC-02, L1-CIC-03; ASM-01–ASM-07

The schema shall represent formative and summative assessments, supported item types, answer and grading definitions, question pools, selection behavior, thresholds, attempts, time limits, feedback release, rationales, and mappings to learning objectives/systems/subsystems. Protected answer data shall be distinguishable from learner-visible data.

**Acceptance criteria**

- **GIVE** an assessment fixture containing every supported question type and a controlled feedback policy
- **WHEN** the fixture validates
- **THEN** every item has a deterministic grading method, objective mapping, versionable identity, rationale, and valid answer-release configuration

### L2-CIC-07 — Provenance and uncertainty

**Traces to:** L1-CIC-02; AGT-03/04

Generated claims, explanations, objectives, and assessment rationales shall support source citations where direct sources exist. Package-level and element-level metadata shall represent confidence, assumptions, omissions, conflicting sources, and unresolved questions without presenting them as verified facts.

**Acceptance criteria**

- **GIVE** a generated lesson containing a factual architecture claim with a direct supplied source
- **WHEN** the package validates under publication-ready rules
- **THEN** the claim resolves to provenance within the declared source snapshot or produces the configured blocking/warning issue

### L2-CIC-08 — Stable identifiers and references

**Traces to:** L1-CIC-04, L1-CIC-06; AGT-05, CUR-11

All reusable entities shall have stable package-local identifiers following the documented character, length, uniqueness, and normalization rules. References shall target declared identifiers of compatible types. Identifier generation guidance shall allow regeneration from equivalent source scope to retain identities where semantic entities are unchanged.

**Acceptance criteria**

- **GIVE** two packages regenerated from the same stable systems and lessons with revised prose
- **WHEN** their identifiers are compared
- **THEN** unchanged semantic entities retain identifiers and can be paired without title-based guessing

- **GIVE** a duplicate identifier or reference to a missing/incompatible entity
- **WHEN** validation runs
- **THEN** a blocking issue reports the stable code and every relevant JSON path

### L2-CIC-09 — Canonical primitive formats

**Traces to:** L1-CIC-04

The ICD shall define UTF-8 encoding, JSON media type, canonical date/time and timezone representation, duration format, checksum representation, identifier normalization, supported locales, enumerations, null semantics, and default behavior. Defaults shall be applied only where the ICD explicitly permits them.

**Acceptance criteria**

- **GIVE** semantically equivalent package data encoded with documented canonical primitives
- **WHEN** compliant validators process it
- **THEN** they agree on interpreted values and on any normalized representation used for checksums or comparison

### L2-CIC-10 — Validation issue contract

**Traces to:** L1-CIC-03; CUR-02–CUR-04

Each validation issue shall include a stable code, severity, publication-blocking classification, human-readable explanation, precise JSON Pointer or equivalent path, and optional safe remediation guidance. Error order shall be deterministic. Values that may contain source, answer, secret, or personal data shall not be copied into issue messages by default.

**Acceptance criteria**

- **GIVE** an invalid fixture with a duplicate ID, dangling reference, unsupported block, and missing rationale
- **WHEN** validation runs twice with the same validator release
- **THEN** both runs return the same ordered codes, severities, blocking states, and paths without sensitive payload values

### L2-CIC-11 — Size and complexity limits

**Traces to:** L1-CIC-04, L1-CIC-07

The contract and validator shall define enforceable limits for package bytes, entity counts, string/excerpt length, assessment items, reference counts, and nesting depth. Limit failures shall occur before unsafe resource consumption and shall distinguish unsupported pilot scale from malformed content.

**Acceptance criteria**

- **GIVE** a package that exceeds the configured nesting or entity-count limit
- **WHEN** validation receives it
- **THEN** processing stops within the documented resource guardrail and returns a safe blocking limit issue without partially importing content

### L2-CIC-12 — Version compatibility and migration

**Traces to:** L1-CIC-05

The ICD shall define semantic-version meaning, supported-version window, forward/backward compatibility, deprecation notice period, migration responsibility, and behavior for unsupported major/minor versions. A migration shall preserve stable identifiers and protected semantics or report any unavoidable loss.

**Acceptance criteria**

- **GIVE** a package using a supported older contract version
- **WHEN** RepoFluent accepts or migrates it
- **THEN** the resulting draft identifies both source and effective versions, preserves stable identities, and reports all transformations and warnings

### L2-CIC-13 — Extension mechanism

**Traces to:** L1-CIC-06

Extensions shall use a documented namespace mechanism, shall not redefine core fields, and shall be ignorable or rejectable according to declared criticality. Unsupported critical extensions shall block import; unsupported noncritical extensions may produce a warning without changing core interpretation.

**Acceptance criteria**

- **GIVE** a valid package with one unsupported noncritical extension and one unsupported critical extension in separate test cases
- **WHEN** validation runs
- **THEN** the noncritical case preserves core content with a warning and the critical case is blocked with an extension-specific issue

### L2-CIC-14 — Conformance fixture suite

**Traces to:** L1-CIC-08

Every contract release shall include a minimal valid fixture, a representative C#/Angular curriculum, and invalid fixtures covering required fields, types, identifiers, references, ordering, security, assessment rules, and limits. Fixtures shall declare their expected issue codes or successful outcome.

**Acceptance criteria**

- **GIVE** a candidate contract or validator release
- **WHEN** the full fixture suite runs in a clean environment
- **THEN** valid fixtures pass, invalid fixtures produce their declared issues, and any changed outcome blocks release until reviewed and versioned
