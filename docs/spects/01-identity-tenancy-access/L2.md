## Detailed requirements

### L2-ITA-01 — Mandatory tenant context

**Traces to:** L1-ITA-01; IAM-01

Every authenticated principal, session, service credential, persisted identity record, and authorization request shall carry a server-resolved tenant identifier. Repositories and query services shall apply the tenant predicate independently of client-supplied filters. Cross-tenant identifiers shall be treated as nonexistent to the requesting actor.

**Acceptance criteria**

- **GIVE** two tenants with users and records that use different platform identifiers
- **WHEN** a user from the first tenant requests a known record identifier owned by the second tenant
- **THEN** the operation is denied without returning the record, its metadata, or confirmation that it exists, and the denial is recorded with a correlation identifier

### L2-ITA-02 — Local invitation lifecycle

**Traces to:** L1-ITA-02; IAM-02

An authorized administrator shall be able to issue a single-tenant, single-use invitation with an expiry. Acceptance shall verify the intended identity, require supported credential controls, activate the membership once, and invalidate the invitation. Expired, revoked, reused, or tenant-mismatched invitations shall fail safely.

**Acceptance criteria**

- **GIVE** a valid unexpired invitation issued for a user in tenant A
- **WHEN** the intended user completes the invitation flow successfully
- **THEN** one active tenant-A membership is created, the invitation cannot be reused, and invitation and activation audit events identify the actor and time

### L2-ITA-03 — Enterprise single sign-on

**Traces to:** L1-ITA-02; IAM-02

The production authentication path shall validate identity-provider signatures, issuer, audience, lifetime, replay protections, and tenant routing before establishing a RepoFluent session. Claims-to-user and claims-to-group mappings shall use an administrator-approved configuration and shall not grant unsupported roles implicitly.

**Acceptance criteria**

- **GIVE** a configured tenant identity provider and a correctly signed, current assertion for an assigned user
- **WHEN** the user signs in through the enterprise path
- **THEN** RepoFluent creates a session for the resolved tenant and user with only mapped memberships and roles

- **GIVE** an assertion with an invalid issuer, audience, signature, lifetime, replay state, or tenant route
- **WHEN** the sign-in endpoint processes it
- **THEN** no session is created, the user receives a non-disclosing failure, and security telemetry records the failure category without assertion contents

### L2-ITA-04 — Role and scope policy evaluation

**Traces to:** L1-ITA-03, L1-ITA-08; IAM-03

Authorization shall evaluate the requested action against all actor roles and explicit constraints for tenant, team, curriculum, system, and repository classification. Multiple roles shall combine grants but shall not override an explicit resource restriction. Server-side enforcement shall be authoritative; UI visibility is a convenience, not a security control.

**Acceptance criteria**

- **GIVE** a user who is both a learner and a manager for team Alpha, with no grant for team Beta
- **WHEN** the user views team analytics and directly calls the analytics API
- **THEN** Alpha data is available under the manager grant, Beta data is denied at the API, and learner permissions remain available without broadening manager scope

### L2-ITA-05 — Role capabilities

**Traces to:** L1-ITA-03; PRD 6.2

The baseline policy set shall distinguish Learner, Author, Reviewer, Manager, Administrator, and Auditor capabilities as defined by the PRD. Auditor access shall remain read-only. Reviewer approval capability shall not imply administrative publication capability. Manager access shall be limited to explicitly authorized populations.

**Acceptance criteria**

- **GIVE** one user assigned to each baseline role and a representative protected operation for every role
- **WHEN** the authorization conformance suite executes allowed and prohibited operations
- **THEN** each role receives only its documented capabilities, every prohibited operation is denied server-side, and the policy version is included in test evidence

### L2-ITA-06 — Group and team membership

**Traces to:** L1-ITA-04; IAM-04

Authorized administrators shall create, rename, deactivate, and manage tenant-local groups and teams. Membership changes shall be effective for new authorization and assignment operations within the documented propagation target. Deactivation shall preserve historical assignment and audit references.

**Acceptance criteria**

- **GIVE** an active group assigned a curriculum and an administrator with group-management permission
- **WHEN** the administrator adds a learner to the group
- **THEN** the learner receives the derived assignment once, membership and assignment provenance identify the group, and no other tenant is affected

### L2-ITA-07 — Identity audit evidence

**Traces to:** L1-ITA-05; IAM-05

The subsystem shall emit immutable audit events for invitations, sign-in security events, sign-out or revocation, user activation/deactivation, group membership changes, role grants/removals, policy changes, and provisioning results. Events shall include tenant, actor or system principal, subject, action, result, timestamp, and correlation identifier.

**Acceptance criteria**

- **GIVE** an administrator changes a user from Learner to Author
- **WHEN** the change commits
- **THEN** the effective role changes atomically and an immutable audit event records the prior value, new value, actor, subject, tenant, time, and correlation identifier

### L2-ITA-08 — Automated provisioning

**Traces to:** L1-ITA-06; IAM-06

The provisioning interface should support idempotent creation, update, activation, deactivation, and group membership synchronization using a selected enterprise provisioning standard. External identifiers shall be tenant-scoped. Replayed requests shall converge on the same state without duplicate users or memberships.

**Acceptance criteria**

- **GIVE** a provisioning create request with a stable external identifier that has already succeeded
- **WHEN** the identity provider retries the same request
- **THEN** the existing tenant user is returned or updated consistently, no duplicate is created, and both attempts are traceable

### L2-ITA-09 — Session and revocation controls

**Traces to:** L1-ITA-07; IAM-07

Supported tenant policies should configure session idle timeout, absolute lifetime, reauthentication for sensitive operations, and administrative session revocation within platform safety limits. Revoked or expired sessions shall fail on the next protected request and shall not be refreshable.

**Acceptance criteria**

- **GIVE** a user with an active session and an administrator authorized to revoke it
- **WHEN** the administrator revokes the session
- **THEN** the next protected browser or API request is rejected, reauthentication is required, and the revocation is audited

### L2-ITA-10 — Access review evidence

**Traces to:** L1-ITA-07, L1-ITA-08; IAM-07

The subsystem should provide an authorized, tenant-scoped view or export of users, status, roles, group memberships, direct grants, constrained scopes, last access, and review state. Export shall follow report-export authorization and audit controls.

**Acceptance criteria**

- **GIVE** an administrator starts an access review for tenant A
- **WHEN** the administrator retrieves the review dataset
- **THEN** it contains current tenant-A access grants and their provenance, contains no tenant-B identity data, and the retrieval or export is audited

### L2-ITA-11 — Secure failure and availability behavior

**Traces to:** L1-ITA-09

Missing policy data, unavailable authorization dependencies, malformed identifiers, or inconsistent tenant context shall fail closed for protected operations. User-facing errors shall use safe identifiers; operational telemetry shall permit diagnosis without storing credentials or protected token material.

**Acceptance criteria**

- **GIVE** the authorization policy dependency is unavailable during a protected write
- **WHEN** an authenticated client attempts the write
- **THEN** the write does not occur, the client receives a retry-safe non-disclosing error identifier, and operational telemetry captures the dependency and correlation identifiers
