## Detailed requirements

### L2-EXP-01 — Design-token contract

**Traces to:** L1-EXP-01

The design system shall define named semantic tokens for surfaces, text, borders, focus, status, charts, typography, spacing, sizing, radius, elevation, motion duration/easing, and layering. Components shall consume semantic tokens rather than tenant-specific raw values. Token versions and breaking-change policy shall be documented.

**Acceptance criteria**

- **GIVE** default and tenant-branded themes
- **WHEN** the shared component conformance suite renders launch-critical states
- **THEN** components use declared semantic tokens, required contrast/focus rules pass in both themes, and no domain status relies on an undocumented raw color

### L2-EXP-02 — Component state coverage

**Traces to:** L1-EXP-01, L1-EXP-06

Interactive primitives shall define default, hover, focus-visible, active, selected, disabled, busy, success, warning, error, and read-only states as applicable. State shall be conveyed through text/semantics or shape/iconography in addition to color.

**Acceptance criteria**

- **GIVE** a keyboard user traverses controls in each applicable state under high zoom
- **WHEN** focus and state change
- **THEN** the current focus/state is visible, programmatically exposed, not color-only, and does not cause content loss or overlap

### L2-EXP-03 — Motion and reduced motion

**Traces to:** L1-EXP-02, L1-EXP-06

Motion shall be short, interruptible, nonblocking, and used only to explain transitions/state/relationships. The platform shall honor `prefers-reduced-motion` and any supported user setting by removing nonessential movement and replacing spatial animation with immediate or low-motion state change.

**Acceptance criteria**

- **GIVE** reduced motion is enabled
- **WHEN** a user navigates, opens a drawer, changes progress, or selects a map node
- **THEN** the state change remains clear and operable without parallax, long transition, auto-pan, or comprehension depending on animation

### L2-EXP-04 — Progressive GPU capability

**Traces to:** L1-EXP-03; pilot AC-8

GPU initialization shall be capability-detected, time-bounded, and isolated from core rendering. Failure, device loss, unsupported browser, or policy disablement shall select an equivalent non-GPU visualization/transition without blocking page content or controls.

**Acceptance criteria**

- **GIVE** WebGPU is unsupported or initialization throws
- **WHEN** a learner opens the home, lesson, assessment, and system map
- **THEN** all required content and operations load through the fallback, a safe capability event is recorded, and no answer/status/navigation is missing

### L2-EXP-05 — Accessible visualization companion

**Traces to:** L1-EXP-03, L1-EXP-06

Every interactive canvas/GPU/SVG graph shall have a semantic companion list/table/tree that exposes equivalent nodes, relationships, directions, status, selection, details, and actions. Selection and filter state shall remain synchronized where both presentations coexist.

**Acceptance criteria**

- **GIVE** a system relationship map with seven nodes and typed edges
- **WHEN** a screen-reader user navigates and selects through the companion representation
- **THEN** all nodes/edges/details/actions are available and selection matches the visual view without requiring pointer or canvas access

### L2-EXP-06 — Context-preserving navigation primitives

**Traces to:** L1-EXP-04

The platform shall provide route, breadcrumb, tab, drawer, split-pane, focus-return, and back-navigation patterns that retain domain context in URL/state where safe. Opening contextual code, glossary, or map detail shall not discard unsaved/progress state.

**Acceptance criteria**

- **GIVE** a learner at a specific lesson section opens a glossary drawer and a code split view
- **WHEN** each context is closed and browser Back is used according to the route model
- **THEN** focus and reading position return predictably and no duplicate progress/navigation entry corrupts history

### L2-EXP-07 — Responsive layout profiles

**Traces to:** L1-EXP-05

Supported profiles shall define minimum viewport, zoom, pointer, and orientation expectations for desktop, tablet, and narrow use. Navigation, assessment, lesson, import/review, administration, and analytics shall reflow without two-dimensional page scrolling, clipped controls, or inaccessible offscreen actions except intentional code/table regions with their own scrolling.

**Acceptance criteria**

- **GIVE** every launch-critical flow at each supported viewport and 200% zoom
- **WHEN** representative tasks are completed by keyboard
- **THEN** content reflows, controls remain visible/operable, focus is not obscured, and only declared inner regions require horizontal scrolling

### L2-EXP-08 — Semantic structure and accessible names

**Traces to:** L1-EXP-06

Pages shall use meaningful landmarks/headings, native semantics where possible, unique accessible names, labels/instructions, described errors, table headers, and relationship semantics. Custom widgets shall implement required keyboard patterns and state properties.

**Acceptance criteria**

- **GIVE** automated accessibility tests and manual screen-reader review of the learner, assessment, import/review, and analytics flows
- **WHEN** users navigate by landmarks, headings, controls, and tables
- **THEN** structure and names communicate purpose/order/state with no critical violation or keyboard trap

### L2-EXP-09 — Focus management

**Traces to:** L1-EXP-06

Focus order shall follow visual/logical reading order. Dialogs/drawers shall move, contain where modal, and restore focus. Route changes shall announce or focus the new primary heading per pattern. Busy updates shall not steal focus. Sticky UI shall not obscure the focused element.

**Acceptance criteria**

- **GIVE** keyboard focus invokes a modal and then closes it by action or Escape
- **WHEN** the interaction completes
- **THEN** focus is never lost, remains contained while modal, returns to the invoker, and visible focus is not obscured

### L2-EXP-10 — Status, errors, and live announcements

**Traces to:** L1-EXP-06

Async save, validation, grading submission, processing, success, warning, and error states shall use persistent visible status appropriate to severity and programmatic live announcements without excessive repetition. Errors shall identify the affected control/entity and recovery action.

**Acceptance criteria**

- **GIVE** an assessment submission succeeds after a retryable network interruption
- **WHEN** status changes from saving to retrying to submitted
- **THEN** sighted and screen-reader users receive accurate nonduplicative state, the submit action is not repeated, and focus remains usable

### L2-EXP-11 — Text alternatives and media

**Traces to:** L1-EXP-06

Meaningful diagrams/images shall provide concise alternatives and extended descriptions where needed. Decorative visuals shall be hidden from assistive technology. If future audio/video is supported, captions/transcripts and player accessibility shall be required before launch.

**Acceptance criteria**

- **GIVE** a lesson diagram whose relationships are necessary to meet an objective
- **WHEN** images or graphics are unavailable
- **THEN** an equivalent description/relationship representation communicates the necessary learning content

### L2-EXP-12 — Learner-shell performance budget

**Traces to:** L1-EXP-07; PRD 9.3

The production measurement profile shall define device, browser, connection, cache, tenant content envelope, and usability milestone. At p75, the learner shell shall meet 2.5 seconds excluding explicitly governed unusually large optional assets. Measurement shall use real-user monitoring plus repeatable lab gates.

**Acceptance criteria**

- **GIVE** the approved representative production profile and content fixture
- **WHEN** release lab tests and production p75 monitoring evaluate shell usability
- **THEN** the milestone is at or below 2.5 seconds or release/operations policy records and addresses a budget breach

### L2-EXP-13 — Interaction and animation budgets

**Traces to:** L1-EXP-08

Typical interactions shall meet the p75 200 ms target from user input to visible response under the defined profile. Animations shall target 60 fps, avoid long main-thread tasks, and degrade effects before blocking input or semantic updates.

**Acceptance criteria**

- **GIVE** representative navigation, search filter, drawer, progress, and map-selection interactions
- **WHEN** performance testing measures latency and frames under the approved profile
- **THEN** p75 interactions meet 200 ms, animation frames meet the target within tolerance, and any effect degradation preserves functionality

### L2-EXP-14 — Large-content rendering

**Traces to:** L1-EXP-09

Long lessons, file trees, tables, result lists, and analytics datasets shall use an appropriate paging/virtualization/progressive strategy that preserves semantic order, focus, screen-reader access, find/filter behavior, and stable scroll position.

**Acceptance criteria**

- **GIVE** each large-content fixture at the supported pilot limit
- **WHEN** a keyboard/screen-reader user navigates, filters, and returns to a selected item
- **THEN** interaction remains within budget, focus/position remain stable, and all permitted records are reachable without rendering the entire dataset at once

### L2-EXP-15 — Browser policy and capability monitoring

**Traces to:** L1-EXP-10

Before launch, the product shall publish supported browser/version/capability profiles and test them in CI/release gates. Production telemetry shall measure capability/fallback use and critical performance/error outcomes without fingerprinting beyond approved need. Unsupported browsers shall receive accurate guidance while preserving safe access where feasible.

**Acceptance criteria**

- **GIVE** the approved browser matrix and a release candidate
- **WHEN** cross-browser gates run
- **THEN** all critical flows pass on supported profiles, fallback profiles pass without GPU, and any unsupported profile receives documented non-disclosing guidance
