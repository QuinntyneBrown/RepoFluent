## Detailed requirements

### L2-CLI-01 — Authorized upload intake

**Traces to:** L1-CLI-01, L1-CLI-11; CUR-01/05

Only an authorized Author or stronger explicit grant shall initiate upload for a tenant. Intake shall enforce media type and configured byte limits, compute a checksum, scan for malformed/unsafe content, assign an opaque platform identifier, and persist a tenant-scoped receipt before asynchronous processing.

**Acceptance criteria**

- **GIVE** an authorized author and a package within supported type and size limits
- **WHEN** upload completes
- **THEN** the author receives a tenant-scoped receipt and correlation identifier, the checksum is recorded, validation is queued once, and no learner can resolve the content

### L2-CLI-02 — Intake rejection

**Traces to:** L1-CLI-01, L1-CLI-02

Unsupported types, limit violations, malformed archives, executable payloads, and failed security scans shall be rejected before import. Rejection shall not create learner-visible or partially imported entities and shall provide a safe actionable result to the author.

**Acceptance criteria**

- **GIVE** an upload containing an executable payload or exceeding the configured package limit
- **WHEN** intake processes the upload
- **THEN** import does not begin, no curriculum entities are committed, a blocking issue and safe support identifier are returned, and the event is audited

### L2-CLI-03 — Validation orchestration

**Traces to:** L1-CLI-02; CUR-02

Validation shall execute the supported contract checks for version, schema, identifiers, references, ordering, content types, source references, provenance, assessment integrity, security restrictions, and configured limits. The validation result shall identify validator and contract versions and package checksum.

**Acceptance criteria**

- **GIVE** the representative invalid conformance fixture
- **WHEN** lifecycle validation completes
- **THEN** the stored report includes all expected check categories, versions, checksum, deterministic issue list, and a failed publication gate

### L2-CLI-04 — Validation severity and acknowledgement

**Traces to:** L1-CLI-03; CUR-03/04

Errors shall block draft preview or publication according to the validation stage. Warnings shall require an authorized acknowledgement with actor, time, issue set/checksum, and optional rationale before approval. A changed issue set shall invalidate prior warning acknowledgement.

**Acceptance criteria**

- **GIVE** a draft with no errors and two warnings
- **WHEN** an authorized reviewer acknowledges the exact warning report and the package remains unchanged
- **THEN** the review gate records the acknowledgement and may advance

- **GIVE** the package changes after warning acknowledgement
- **WHEN** it is revalidated
- **THEN** the prior acknowledgement no longer satisfies the gate and the new issue set requires review

### L2-CLI-05 — Atomic draft import

**Traces to:** L1-CLI-01, L1-CLI-09; CUR-05/11

A package that passes import-blocking checks shall be converted into a draft transactionally. Stable package identifiers shall remain separate from platform tenant identifiers. Failed conversion shall not leave a resolvable partial draft and shall be safe to retry.

**Acceptance criteria**

- **GIVE** a valid package whose conversion encounters a transient persistence failure
- **WHEN** the operation retries
- **THEN** exactly one complete draft exists or none exists, no duplicate entities are created, and the lifecycle record shows correlated attempts

### L2-CLI-06 — Learner-equivalent preview

**Traces to:** L1-CLI-04; CUR-06

Preview shall use the same renderer, sanitization, access policy, answer protection, responsive behavior, and fallback behavior as publication, while clearly marking draft/version context. Preview access shall be restricted to authorized review participants and shall not create learner progress or assessment evidence.

**Acceptance criteria**

- **GIVE** an authorized reviewer and a valid draft containing lessons, code references, a map, quiz, and controlled test
- **WHEN** the reviewer opens preview
- **THEN** the content uses production rendering/access behavior, is visibly marked as draft, protected answers remain hidden, and no learner record is written

### L2-CLI-07 — Review decision

**Traces to:** L1-CLI-05; CUR-07

An authorized Reviewer shall approve or reject a specific content checksum and validation report. The decision shall capture reviewer, tenant, version candidate, decision, time, warning acknowledgement, and comment/rationale where required. An author shall not self-approve unless separately granted Reviewer permission by policy.

**Acceptance criteria**

- **GIVE** a reviewer viewing draft checksum A with satisfied validation gates
- **WHEN** the reviewer approves checksum A
- **THEN** the approval is immutable and applies only to A; any later checksum requires a new decision

### L2-CLI-08 — Controlled publication

**Traces to:** L1-CLI-05, L1-CLI-11; CUR-07/08

Only an authorized Administrator shall publish an approved, unchanged version. Publication shall atomically assign the immutable published version identity, activate its availability policy, and emit audit and domain events. Duplicate or concurrent publication commands shall converge on one result.

**Acceptance criteria**

- **GIVE** an approved version and two concurrent identical publication requests
- **WHEN** both requests complete
- **THEN** exactly one immutable published version is active, both callers receive a consistent outcome, and one publication event is recorded

### L2-CLI-09 — Published immutability

**Traces to:** L1-CLI-06; CUR-08

Published content, source snapshot, assessment definitions, and protected answer policies shall be read-only. Corrections shall produce a new draft and published version. Learning progress, attempts, mastery evidence, and audit records shall retain the originating version identifiers.

**Acceptance criteria**

- **GIVE** learners have completed assessments on published version 1
- **WHEN** an author uploads corrected content and version 2 is published
- **THEN** version 1 content/evidence remains unchanged and queryable to authorized users, while new assignments may target version 2

### L2-CLI-10 — Retirement and unpublication

**Traces to:** L1-CLI-07; CUR-09

Retirement shall prevent new discovery or assignment and shall follow configured behavior for existing assignments while preserving content required to interpret historical records. It shall not hard-delete evidence. Reversal, if supported by policy, shall be an explicit audited transition.

**Acceptance criteria**

- **GIVE** a published curriculum with completed and active assignments
- **WHEN** an administrator retires it
- **THEN** it cannot be newly discovered or assigned, historical evidence remains resolvable to authorized users, active-assignment behavior follows the declared policy, and retirement is audited

### L2-CLI-11 — Semantic version comparison

**Traces to:** L1-CLI-08; CUR-10

The subsystem should compare stable entities across versions and classify added, removed, modified, reordered, source-changed, assessment-changed, and unresolved-reference changes. It should identify affected objectives and distinguish presentational edits from changes that may require learner refresh.

**Acceptance criteria**

- **GIVE** a new draft that changes one lesson source, removes one question, reorders a module, and edits unrelated punctuation
- **WHEN** a reviewer opens the comparison
- **THEN** each semantic change is classified and linked to affected entities/objectives, while the punctuation edit is not misrepresented as an assessment-impacting change

### L2-CLI-12 — Idempotent re-import

**Traces to:** L1-CLI-09; CUR-11

Re-upload of identical package identity, version, and checksum should return the existing receipt/draft result within the idempotency window. A reused identity/version with different bytes shall be treated as a conflict requiring a new version or explicit replacement workflow.

**Acceptance criteria**

- **GIVE** a package has already produced a valid draft
- **WHEN** the author uploads the identical bytes again
- **THEN** no duplicate draft/entities are created and the response identifies the existing result

### L2-CLI-13 — Lifecycle status and recovery

**Traces to:** L1-CLI-11

Authors and administrators shall see current state, completed/active stage, timestamps, actionable safe failure details, and support correlation identifier. Stale processing jobs shall be detected. Authorized retry shall resume from a safe stage without bypassing validation or approval.

**Acceptance criteria**

- **GIVE** validation processing stops unexpectedly and exceeds the stale-job threshold
- **WHEN** monitoring and the lifecycle UI evaluate the job
- **THEN** it is marked actionable, an alert/diagnostic is emitted, and an authorized retry does not duplicate the draft or skip gates

### L2-CLI-14 — Lifecycle audit trail

**Traces to:** L1-CLI-11; IAM-05

Upload, scan, validation, acknowledgement, draft import, review, approval/rejection, publication, retry, retirement, and version-comparison access shall produce tenant-scoped audit or operational evidence appropriate to sensitivity.

**Acceptance criteria**

- **GIVE** a curriculum proceeds from upload through retirement
- **WHEN** an Auditor retrieves its history
- **THEN** the ordered record identifies actors/systems, version/checksum, transitions, decisions, timestamps, and correlations without mutable gaps

### L2-CLI-15 — Contract-compatible visual edits

**Traces to:** L1-CLI-10; CUR-12

A future visual editor may modify only supported contract fields, shall validate continuously, shall preserve stable identifiers by default, and shall export a package conforming to a declared contract version. It shall use the same review and publication workflow as uploaded content.

**Acceptance criteria**

- **GIVE** visual authoring is enabled and an author changes supported lesson metadata
- **WHEN** the author exports or submits the edit
- **THEN** the output passes the same contract validator, unchanged entities retain identifiers, and no publication occurs without review and approval
