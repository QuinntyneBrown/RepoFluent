## Detailed requirements

### L2-AAK-01 — Kit contents and release manifest

**Traces to:** L1-AAK-01; AGT-01

Each kit release shall contain `AGENTS.md`, reusable prompts, applicable `SKILL.md` instructions, the compatible JSON Schema and ICD, valid and invalid examples, a local validation command, release notes, artifact checksums, and a manifest declaring kit, contract, and validator versions.

**Acceptance criteria**

- **GIVE** a clean supported workstation with the documented runtime prerequisites
- **WHEN** an author unpacks a released kit and follows its quick start
- **THEN** all declared artifacts are present, checksums pass, and the representative package validates without obtaining undeclared files

### L2-AAK-02 — Repository instruction precedence

**Traces to:** L1-AAK-02; AGT-02

`AGENTS.md` shall instruct the agent to discover and obey applicable repository guidance, resolve scope by directory, preserve explicit exclusions, and stop or report conflict rather than silently overriding higher-priority customer instructions.

**Acceptance criteria**

- **GIVE** a test repository whose root guidance permits analysis but a subsystem guidance file excludes a directory
- **WHEN** a conforming agent generates a package using the kit
- **THEN** excluded content is absent, the generation report identifies the exclusion, and no claim is presented as if the excluded content were analyzed

### L2-AAK-03 — Scope and access declaration

**Traces to:** L1-AAK-02, L1-AAK-09

The workflow shall require an explicit source scope, approved repositories/documents, revision or snapshot, exclusions, output location, and data-handling constraints before analysis. The kit shall not instruct an agent to elevate access, bypass controls, or retrieve undeclared sources.

**Acceptance criteria**

- **GIVE** an author declares two approved repositories and one excluded document set
- **WHEN** generation completes
- **THEN** the source snapshot contains only approved sources, the exclusion is reported, and no undeclared access is attempted by the documented workflow

### L2-AAK-04 — Secret and sensitive-data handling

**Traces to:** L1-AAK-02, L1-AAK-09; PRD 11

Instructions shall require agents to avoid collecting secrets, minimize source excerpts, honor content classification/redaction metadata, and stop/report suspected secret exposure. The validation workflow shall support secret-pattern scanning before package release.

**Acceptance criteria**

- **GIVE** a seeded credential-like value in an otherwise approved source fixture
- **WHEN** package generation and preflight validation run
- **THEN** the value is not emitted, a safe blocking finding is reported, and logs do not reproduce the secret

### L2-AAK-05 — Source citation procedure

**Traces to:** L1-AAK-03; AGT-03

The kit shall define how to create repository-relative and document citations, bind them to the source snapshot, associate them with claims/objectives, and distinguish direct evidence from synthesis or interpretation.

**Acceptance criteria**

- **GIVE** source fixtures that directly describe a service responsibility and code path
- **WHEN** an agent creates the corresponding lesson and objective
- **THEN** both resolve to valid source references and the generation report distinguishes sourced facts from synthesis

### L2-AAK-06 — Uncertainty report

**Traces to:** L1-AAK-04; AGT-04

The authoring workflow shall produce a structured report of assumptions, confidence, conflicting evidence, missing context, omissions, and unresolved questions. Material uncertainty shall also be represented in the package where the learner/reviewer needs it.

**Acceptance criteria**

- **GIVE** two approved documents that conflict about a subsystem dependency
- **WHEN** the agent prepares the curriculum
- **THEN** the conflict is recorded with both sources and is not collapsed into an unqualified factual statement

### L2-AAK-07 — Deterministic identity procedure

**Traces to:** L1-AAK-05; AGT-05

The kit shall prescribe stable namespace inputs, normalization, collision handling, and regeneration behavior for package, system, course, lesson, objective, code-reference, and assessment identifiers. Human-readable titles shall not be the only identity source.

**Acceptance criteria**

- **GIVE** two generation runs over the same stable source scope with only prose wording changes
- **WHEN** their package entities are compared
- **THEN** unchanged semantic entities retain identifiers and any collision is reported rather than overwritten

### L2-AAK-08 — Local validator interface

**Traces to:** L1-AAK-06; AGT-06

The kit shall document a noninteractive local command that accepts package path, contract version or auto-detection, output format, and severity threshold. Exit status shall distinguish success, warnings-only, validation failure, invalid invocation, and internal validator failure. JSON output shall implement the validation issue contract.

**Acceptance criteria**

- **GIVE** valid, warnings-only, and blocking-error fixtures
- **WHEN** each is processed with the documented local command in offline mode
- **THEN** output and exit status match the fixture outcome and no package content leaves the workstation

### L2-AAK-09 — C# analysis guidance

**Traces to:** L1-AAK-07; AGT-07

C# guidance should cover solution/project structure, application boundaries, dependency injection, controllers/endpoints, domain services, persistence, messaging, configuration, background workers, external clients, and tests. It shall tell the agent to report dynamic or unresolved behavior rather than infer it as fact.

**Acceptance criteria**

- **GIVE** the representative C# fixture with API, worker, dependency injection, messaging, persistence, and tests
- **WHEN** a conforming analysis follows the guidance
- **THEN** the package identifies the declared boundaries and source paths, and flags intentionally ambiguous dynamic registration

### L2-AAK-10 — Angular analysis guidance

**Traces to:** L1-AAK-07; AGT-07

Angular guidance should cover application bootstrap, route boundaries, standalone components or modules, services, dependency injection, state flow, HTTP integration, guards/interceptors, templates, configuration, and tests.

**Acceptance criteria**

- **GIVE** the representative Angular fixture with checkout route, components, services, state, HTTP client, guard, and tests
- **WHEN** a conforming analysis follows the guidance
- **THEN** the curriculum describes the user-to-API flow with valid citations and does not invent runtime behavior absent from the fixture

### L2-AAK-11 — Generation manifest

**Traces to:** L1-AAK-08; AGT-08

The workflow should produce a manifest containing tool and model identifiers/versions where available, kit and contract version, source snapshot, generation start/end timestamps, package checksum, declared options, and validation result. It shall exclude hidden chain-of-thought, credentials, and full prompt transcripts unless separately approved.

**Acceptance criteria**

- **GIVE** a completed generation run
- **WHEN** an author inspects its manifest
- **THEN** the package can be associated with tool/version, inputs, time, checksum, and validation result without exposing hidden reasoning or secrets

### L2-AAK-12 — Safe offline operation

**Traces to:** L1-AAK-09; AGT-06

Schema resolution, fixtures, documentation, and local validation shall function without network access after kit acquisition. Any optional network-dependent feature shall be identified, disabled by default for offline validation, and unnecessary for producing a conformant package.

**Acceptance criteria**

- **GIVE** a supported environment with outbound network access disabled and a locally available generation output
- **WHEN** the author validates and reviews the package using the kit
- **THEN** all mandatory checks and documentation remain available and no hidden download is attempted
