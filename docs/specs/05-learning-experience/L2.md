## Detailed requirements

### L2-LEX-01 — Personalized learning home

**Traces to:** L1-LEX-01; LRN-01

The home view shall show current assignments, required/optional state, not-started/in-progress/completed status, due/overdue state, recent durable activity, and one recommended next action. Data shall be tenant- and learner-scoped and shall exclude drafts or inaccessible versions.

**Acceptance criteria**

- **GIVE** a learner with one overdue required course, one in-progress course, one optional course, and recent activity
- **WHEN** the learner opens the home view
- **THEN** each item has the correct distinct state, the activity reflects durable records, and the recommended action links to an authorized resumable location

### L2-LEX-02 — Next-action selection

**Traces to:** L1-LEX-01, L1-LEX-07

The next-action service shall use documented inputs such as requirement, due date, prerequisite state, progress, assessment outcome, and mastery/recency. The UI shall state why the action is recommended. Missing data shall produce a safe deterministic fallback, not an invented urgency.

**Acceptance criteria**

- **GIVE** a due-soon required lesson that is eligible and an optional lesson recently opened
- **WHEN** the next action is calculated
- **THEN** the required lesson is recommended with an accurate due-date reason and the user can inspect the inputs used

### L2-LEX-03 — Curriculum hierarchy and metadata

**Traces to:** L1-LEX-02; LRN-02

The learner renderer shall preserve published course, module, and lesson ordering and display outcomes/objectives, prerequisites, difficulty, estimated duration, audience, and completion rules where supplied. Unsatisfied prerequisites shall be visible with the configured block or advisory behavior.

**Acceptance criteria**

- **GIVE** a published course with ordered modules, a prerequisite, objectives, and estimated durations
- **WHEN** an assigned learner opens the course
- **THEN** the hierarchy and metadata match the published version and the prerequisite state is accurately explained

### L2-LEX-04 — Safe lesson block rendering

**Traces to:** L1-LEX-02, L1-LEX-12

The lesson renderer shall support only allow-listed contract blocks, sanitize display content, provide text alternatives/accessible companions, and fail an individual unsupported block safely without hiding the remainder of an otherwise renderable lesson. The visible error shall carry a support identifier and version context.

**Acceptance criteria**

- **GIVE** a lesson containing valid prose, callout, diagram alternative, glossary reference, and code reference plus one renderer failure
- **WHEN** the learner opens the lesson
- **THEN** valid blocks remain usable, the failed block has an accessible safe error and correlation identifier, and no active content executes

### L2-LEX-05 — Progress event semantics

**Traces to:** L1-LEX-03; LRN-03

The subsystem shall record meaningful progress events including version, learner, lesson/block position, completion evidence, timestamps, client event identity, and relevant completion-rule inputs. Mere page load or scroll shall not automatically satisfy demonstrated completion unless the published rule explicitly permits it.

**Acceptance criteria**

- **GIVE** a lesson whose completion rule requires visiting all required sections and completing a knowledge check
- **WHEN** the learner only opens the page and scrolls to the end
- **THEN** position is saved but completion remains unsatisfied and the remaining conditions are explained

### L2-LEX-06 — Cross-device resume and conflict handling

**Traces to:** L1-LEX-03; LRN-03, pilot AC-5

Progress writes shall be idempotent and ordered using event identity/version semantics. Resume shall select the latest valid meaningful position without allowing a stale device to erase a newer completed state. A learner shall be able to continue on another supported device after successful synchronization.

**Acceptance criteria**

- **GIVE** device A saves section 5 and device B later completes the lesson, after which device A submits its stale section-5 event
- **WHEN** the progress record resolves the events
- **THEN** completion remains durable, event history is not duplicated, and subsequent resume does not regress to an incomplete state

### L2-LEX-07 — Search scope and result model

**Traces to:** L1-LEX-04; LRN-04

Search shall index and return only permitted published systems, subsystems, courses, lessons, glossary terms, and code-reference metadata. Results shall identify type, curriculum/version, context snippet, and target. Authorization changes and retirement shall remove discoverability within the documented propagation target.

**Acceptance criteria**

- **GIVE** matching content in an assigned curriculum, a restricted repository, a draft, and another tenant
- **WHEN** the learner searches the common term
- **THEN** only the authorized published result appears and its link retains the correct curriculum/version context

### L2-LEX-08 — Structured navigation and context preservation

**Traces to:** L1-LEX-04, L1-LEX-05; LRN-04/05

Breadcrumbs, course outline, back navigation, drawers, and split views shall preserve the originating curriculum, lesson, position, and selected relationship/code reference. Deep links shall reestablish authorized context or provide a safe recovery route when it is no longer available.

**Acceptance criteria**

- **GIVE** a learner opens a code reference from lesson 4 and then inspects its subsystem on the system map
- **WHEN** the learner returns to learning
- **THEN** lesson 4 reopens at the originating section with course/version context and progress intact

### L2-LEX-09 — Contextual glossary

**Traces to:** L1-LEX-06; LRN-06

An identified glossary term shall expose its definition, aliases, owning curriculum/version, and related entities in an accessible popover, drawer, or inline expansion. Opening/closing the definition shall preserve reading position and keyboard focus.

**Acceptance criteria**

- **GIVE** keyboard focus is on a glossary term inside a lesson
- **WHEN** the learner opens and closes its definition
- **THEN** the definition is announced accessibly, no page navigation is required, and focus returns to the invoking term

### L2-LEX-10 — Remediation recommendation

**Traces to:** L1-LEX-07; LRN-07

After an incorrect formative answer or low objective-level result, the experience shall recommend permitted published lessons/sections mapped to the affected objective. Recommendations shall not reveal protected answers and shall distinguish required remediation from optional review.

**Acceptance criteria**

- **GIVE** a learner answers an event-delivery question incorrectly and its objective maps to lesson sections 3 and 4
- **WHEN** configured feedback is released
- **THEN** the learner sees an objective-specific review link and rationale without receiving a protected answer before its release point

### L2-LEX-11 — Learning-state distinctions

**Traces to:** L1-LEX-08; LRN-08

Required/optional, completion, pass/fail, numerical result, objective coverage, demonstrated mastery, and stale knowledge shall have separate labels, semantics, and accessible non-color indicators. Completion shall not be presented as mastery, and best score shall not silently replace latest score.

**Acceptance criteria**

- **GIVE** a learner completed a required course but failed its test and has stale mastery evidence
- **WHEN** status is presented on home and course views
- **THEN** completion, failure, requirement, and stale mastery are all shown distinctly in text and assistive semantics

### L2-LEX-12 — Private notes

**Traces to:** L1-LEX-09; LRN-09

Learners should create, edit, and delete notes anchored to an authorized curriculum version and optional lesson/block. Notes shall be visible only to their owner by default, excluded from manager analytics/content search, and handled by configured export/retention rules.

**Acceptance criteria**

- **GIVE** a learner creates a private note on a restricted lesson
- **WHEN** another learner, manager, or content author accesses the lesson and reports
- **THEN** the note is absent, while its owner can retrieve it only while authorized under the documented retention behavior

### L2-LEX-13 — Bookmarks

**Traces to:** L1-LEX-09; LRN-09

Learners should bookmark permitted courses, lessons, blocks, system nodes, and code references. A bookmark shall preserve version and context and shall report retired, removed, or newly inaccessible targets without leaking protected metadata.

**Acceptance criteria**

- **GIVE** a learner bookmarks a code reference and later loses repository classification access
- **WHEN** the learner opens the bookmark list
- **THEN** the target content is not disclosed and the bookmark presents a safe unavailable state that can be removed

### L2-LEX-14 — Spaced review

**Traces to:** L1-LEX-10; LRN-10

Spaced review should use documented recency, performance, mastery, prior reviews, and tenant settings. It shall identify the objective and reason, permit deferral within policy, avoid duplicate concurrent review assignments, and recalculate after new evidence.

**Acceptance criteria**

- **GIVE** an objective whose demonstrated mastery has crossed the configured staleness interval
- **WHEN** the review scheduler runs
- **THEN** one explainable review recommendation is created and is updated or cleared when fresh qualifying evidence arrives

### L2-LEX-15 — Discussion and expert Q&A guardrail

**Traces to:** L1-LEX-11; LRN-11

If enabled, content discussion shall be tenant-controlled, restricted to approved cohorts/content, moderated, auditable, classified as user-generated content, and unable to expose protected answers or inaccessible source material. Private notes shall never be converted to discussion without explicit learner action.

**Acceptance criteria**

- **GIVE** discussion is disabled for a tenant or content item
- **WHEN** a learner views or directly requests its discussion endpoint
- **THEN** no discussion content or metadata is available and core learning remains unaffected

### L2-LEX-16 — Core fallback and accessibility

**Traces to:** L1-LEX-12; PRD 9

Home, course/lesson, progress, search, glossary, navigation, and remediation shall function without WebGPU and with reduced motion. Interactive visuals shall provide keyboard operation and an accessible companion representation. Responsive layouts shall support defined desktop, tablet, and narrow profiles.

**Acceptance criteria**

- **GIVE** WebGPU is unavailable, reduced motion is enabled, and the learner uses keyboard-only navigation on a narrow supported viewport
- **WHEN** the learner resumes a lesson, opens glossary/code context, and records progress
- **THEN** every task completes without hidden hover/color/motion-only information and progress is saved successfully
