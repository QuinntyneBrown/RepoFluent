## Detailed requirements

### L2-ASM-01 — Supported item behavior

**Traces to:** L1-ASM-01; ASM-01

The delivery and grading model shall support the six PRD item types. It shall define selection cardinality, order significance, matching cardinality, normalization, case/whitespace behavior for short answer, partial-credit policy, unanswered handling, and deterministic score calculation as versioned item data.

**Acceptance criteria**

- **GIVE** a conformance assessment containing every supported item type and boundary responses
- **WHEN** each response is submitted repeatedly against the same item versions
- **THEN** the normalized response, awarded points, correctness, and feedback eligibility are identical and match the declared grading rules

### L2-ASM-02 — Unsupported or nondeterministic items

**Traces to:** L1-ASM-01

Items lacking a supported deterministic grading definition shall not be included as scored launch assessments. An unsupported item shall block publication or be explicitly represented as unscored enrichment where allowed by the contract.

**Acceptance criteria**

- **GIVE** a short-answer item with no declared acceptable-answer or deterministic grading method
- **WHEN** publication validation runs
- **THEN** the scored assessment is blocked at the item path and cannot be delivered as a gradable question

### L2-ASM-03 — Formative feedback

**Traces to:** L1-ASM-02; ASM-02

Formative quizzes shall support configured immediate or end-of-quiz feedback, correctness visibility, rationale release, retry behavior, and remediation links. The response payload shall include only data authorized at the current release state.

**Acceptance criteria**

- **GIVE** a formative item configured for immediate rationale and remediation
- **WHEN** the learner submits an incorrect response
- **THEN** correctness, permitted rationale, and mapped review content are shown immediately without exposing unrelated protected answers

### L2-ASM-04 — Summative feedback control

**Traces to:** L1-ASM-02, L1-ASM-06; ASM-02/06

Summative tests shall support feedback release after submission, after attempt-window closure, after manual release, or never, as configured. Score, item correctness, answer, and rationale may have independent release rules. All delivery endpoints shall enforce the same server-side state.

**Acceptance criteria**

- **GIVE** a submitted test configured to withhold item correctness and rationales until manual release
- **WHEN** the learner requests results through UI or direct API before release
- **THEN** only currently permitted summary data is returned and protected fields are absent, not merely hidden in the UI

### L2-ASM-05 — Pre-attempt rule disclosure

**Traces to:** L1-ASM-07; ASM-07

Before starting, the learner shall see assessment purpose, formative/summative type, item or pool count, selection behavior disclosure where appropriate, passing/completion rule, total points, attempt limit/remaining attempts, time limit, save/submission behavior, and feedback-release policy.

**Acceptance criteria**

- **GIVE** a timed summative test with two remaining attempts and delayed feedback
- **WHEN** the learner reaches the start gate
- **THEN** all material rules are presented accessibly and the timer/attempt is not consumed until the learner explicitly starts

### L2-ASM-06 — Eligibility and attempt start

**Traces to:** L1-ASM-03, L1-ASM-11; ASM-03

The server shall verify assignment/access, curriculum version, prerequisite/completion eligibility, attempt availability, and active-attempt state before atomically creating an attempt. Repeat start requests with the same idempotency key shall return the same attempt.

**Acceptance criteria**

- **GIVE** an eligible learner sends two concurrent start requests for the final available attempt using one idempotency key
- **WHEN** both requests complete
- **THEN** exactly one active attempt exists, only one attempt entitlement is consumed, and both responses identify it

### L2-ASM-07 — Question-pool selection

**Traces to:** L1-ASM-03; ASM-03

Selection shall honor pool filters, requested counts, objective/coverage constraints, exclusions, and configured deterministic seed or approved randomization. The selected immutable item versions and their order shall be stored at attempt start so retries cannot change an active test.

**Acceptance criteria**

- **GIVE** a pool requiring five items covering three objectives with a stored selection seed
- **WHEN** the attempt is resumed or submission is retried
- **THEN** the same five versioned items and order are used and required objective coverage is satisfied

### L2-ASM-08 — Time-limit enforcement

**Traces to:** L1-ASM-03, L1-ASM-11; ASM-03

The server shall establish authoritative start/deadline times, account for documented accommodations, display remaining time accessibly, warn at configured thresholds, and accept/reject/auto-submit at expiry according to policy. Client clock manipulation shall not extend an attempt.

**Acceptance criteria**

- **GIVE** a ten-minute attempt and a client whose local clock changes during the test
- **WHEN** the server deadline is reached
- **THEN** the configured timeout behavior executes once using received responses and the attempt cannot accept late changes

### L2-ASM-09 — Response save and submission

**Traces to:** L1-ASM-05, L1-ASM-11; ASM-05

Response saves and final submission shall use attempt/item versions, idempotent event identifiers, optimistic concurrency, and durable acknowledgement. Final submission shall atomically close the attempt and enqueue or perform grading once. A closed attempt shall reject response mutation.

**Acceptance criteria**

- **GIVE** a learner double-submits because the first response acknowledgment is lost
- **WHEN** the duplicate submission reaches the service
- **THEN** one immutable submission and grading result exist and both client calls resolve consistently

### L2-ASM-10 — Deterministic grading and outcome

**Traces to:** L1-ASM-01, L1-ASM-03, L1-ASM-05

Grading shall use stored item versions, points/partial-credit definitions, threshold and grading-policy version. The attempt result shall record raw/possible points, percentage or declared scale, pass/fail if applicable, per-objective contribution, grading time, and algorithm/policy version.

**Acceptance criteria**

- **GIVE** a submitted attempt with known responses and a 75% passing threshold
- **WHEN** grading is replayed using the stored versions and policy
- **THEN** raw score, scaled score, pass/fail, and objective contributions reproduce exactly

### L2-ASM-11 — Immutable attempt record

**Traces to:** L1-ASM-05; ASM-05

Attempt evidence shall include tenant, learner, assignment, curriculum/version, assessment/item versions, selected order, response records, start/deadline/submission/grade timestamps, score/outcome, attempt number, feedback state, and correlation/audit references. Corrections shall append superseding evidence, not edit the original.

**Acceptance criteria**

- **GIVE** a completed attempt and a later item invalidation
- **WHEN** authorized users inspect the record before and after recalculation
- **THEN** the original attempt remains intact and the recalculated outcome is linked as a new, attributable derived record

### L2-ASM-12 — Answer isolation

**Traces to:** L1-ASM-06; ASM-06

Protected answer keys, scoring matchers, unreleased rationales, and protected pool content shall be logically separated from learner delivery models, caches, client bundles, search indexes, logs, and pre-release APIs. Reviewer access shall be explicit and audited.

**Acceptance criteria**

- **GIVE** a learner inspects network responses, cached content, page source, and permitted search before answer release
- **WHEN** the security conformance test examines all learner surfaces
- **THEN** protected answers/matchers/rationales are absent and direct protected endpoints deny access

### L2-ASM-13 — Objective mapping and coverage

**Traces to:** L1-ASM-04; ASM-04

Each scored item shall resolve to one or more published objective/system/subsystem identifiers with declared contribution weights where needed. Assessment validation shall report missing mappings, mappings to inaccessible/missing targets, and objectives expected but not covered.

**Acceptance criteria**

- **GIVE** an assessment whose one scored item has no objective mapping and one required objective has no items
- **WHEN** publication quality validation runs
- **THEN** it reports the unmapped item and uncovered objective at precise paths with configured blocking/warning severity

### L2-ASM-14 — Defective-item invalidation

**Traces to:** L1-ASM-08; ASM-08

An authorized Reviewer should invalidate a specific item version with reason and effective scope. Recalculation shall exclude or reweight it under a documented policy, create new result/mastery versions, notify affected consumers as configured, and preserve original evidence.

**Acceptance criteria**

- **GIVE** a defective item affected twenty submitted attempts
- **WHEN** an authorized reviewer invalidates that version and starts recalculation
- **THEN** all and only affected outcomes are recalculated consistently, original outcomes remain auditable, and changed learner-visible results identify the correction

### L2-ASM-15 — Assessment quality checks

**Traces to:** L1-ASM-09; ASM-09

Quality checks should detect exact/near duplicate stems and answers, answer text leaked into visible prompts/explanations, missing rationales, invalid distractor structure, implausible scoring, and objective coverage gaps. Findings shall be deterministic, reviewable, and severity-configurable within product bounds.

**Acceptance criteria**

- **GIVE** seeded fixtures for duplicate questions, answer leakage, missing rationale, and uncovered objective
- **WHEN** quality analysis runs
- **THEN** each seeded defect is reported with issue type and item/objective location and no protected answer is exposed to unauthorized viewers

### L2-ASM-16 — Explainable mastery calculation

**Traces to:** L1-ASM-10; PRD 8.7

Mastery shall be derived per objective using a documented tenant-consistent policy whose inputs may include completion, coverage, latest/best result, attempts, confidence, and recency. The stored record shall identify evidence, weights/rules, calculation version, value/state, and calculation time. Insufficient evidence shall remain distinguishable from low mastery.

**Acceptance criteria**

- **GIVE** two learners, one with no evidence and one with recent failed objective evidence
- **WHEN** mastery is calculated and displayed
- **THEN** the first is `insufficient evidence`, the second reflects low demonstrated mastery, and authorized users can inspect the evidence and calculation version for each

### L2-ASM-17 — Mastery recalculation triggers

**Traces to:** L1-ASM-10, L1-ASM-11

Mastery shall recalculate idempotently after qualifying progress/result evidence, item invalidation, policy-version change, and scheduled recency transition. Recalculation shall not rewrite source evidence and shall prevent an older job from superseding a newer calculation basis.

**Acceptance criteria**

- **GIVE** a newer assessment result arrives while an older mastery recalculation is delayed
- **WHEN** both calculations attempt to persist
- **THEN** the final mastery uses the newest complete evidence basis and retains trace links to both calculation attempts
