# ORDER PROCESSING FOUNDATIONS

How an order becomes a workflow

The API accepts the request quickly. The real work begins after it publishes a message.

OBJECTIVES

[✓] Trace the path from HTTP request to durable order

[✓] Explain why processing happens asynchronously

01. From request to intent

When the storefront submits checkout, validates the command and creates a lightweight order record. It does not reserve stock or charge a card inside the request.

Instead, it publishes OrderSubmitted to the message bus. This creates a durable boundary between accepting the customer’s intent and coordinating downstream work.

// DOMAIN_INSIGHT

“The controller is deliberately boring: validate, persist, publish, return.”

02. A workflow can wait

The consumes the event independently. A temporary inventory outage can delay fulfillment without holding the browser request open or losing the order.

INFO

The message is the hand-off point. Everything before it is synchronous; everything after it is recoverable workflow.
